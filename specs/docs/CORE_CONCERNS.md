# Core Code Concerns — From Requirements Analysis

Generated by cross-referencing 125 impl/ requirements files against the core implementation (sequence.ts, compose.ts, type.ts, statement.ts).

## Status

### FIXED (mechanical, no design decisions needed)

**C2: O(n²) resume/while scanning** — Added `blockBySeq` (Map<seq, Block>) and `suspendedByPath` (Map<path, Set<seq>>) indexes. Resume now scans only candidate blocks for the changed path. While-check uses O(1) map lookup instead of `blocks.find()`. Compact rebuilds indexes.

**C4: Forward-only depIndex** — Added `reverseDepIndex` (Map<dependent, Set<sources>>) to Projection. Maintained alongside the forward index in `applyEntry`. Enables backlink queries (contextgraph/backlinks and 6 other files).

**C6: No block provenance** — Added optional `author?: string` to Block and BlockOpts. Flows through via spread in mount(). Enables permissions/auditability, obligations/trace, and 7 other files.

### FIXED (design decision: Option B)

**C1: Capabilities always external**

- **Was**: `Projection.capabilities: Map<string, Function>` contained closures, blocking serialization.
- **Now**: `Projection.capabilities: Map<string, true>` — serializable markers only. Live functions go to `Sequence.implRegistry` (private, runtime-only). `fireLaws` checks implRegistry first; if absent, yields `PendingInvocation` in MountResult. Caller provides results via `mount('bind', targetPath, result)`.
- **Backward compatible**: `mount('cap', path, fn)` still works — function goes to implRegistry, marker goes to projection. Tests access implementations via `capabilityAt()` instead of `projection.capabilities.get()`.
- **New type**: `PendingInvocation = { capId, args, targetPath }` on MountResult.
- **For fully external invocation**: pass a non-function value (or `true`) to `mount('cap')`, handle `pendingInvocations` from MountResult, provide results via `mount('bind')`.

### FIXED (already implemented — misread on initial analysis)

**C3: Conjunction tracking infrastructure**

- **Was listed as**: missing. Actually already implemented on the Sequence (private fields, not on Projection).
- **What exists**: `conjIndex: Map<string, Set<ConjunctionRef>>`, `conjProbCache`, `gapPriorityCache`. Populated in `indexBlock()`, cleaned in `unindexSuspended()`, delta-propagated via `propagateConjDelta()` called per changed path in `fireLaws`. `gaps()` reads from cache, falls back to computation on miss only.

### NEEDS DESIGN DECISION

### FIXED (implemented + 26 tests)

**C7: Predicate language extended**

- **Was**: evalConstraint handled 9 ops (eq, neq, lt, lte, gt, gte, exists, notExists, count_lt). 15+ requirements files needed richer predicates.
- **Now**: 20 ops. Added composites (or_clause, and_clause, not_clause — recursive, nestable), value predicates (regex, between, one_of, contains, matches_type), and count_gte.
- **Infrastructure**: `Sequence.collectPaths()` recursively extracts path refs from composite predicate trees, so suspendedByPath and conjIndex work correctly with nested predicates.
- **Constructors in type.ts**: `or()`, `and()`, `not()`, `regex()`, `between()`, `oneOf()`, `contains()`, `matchesType()`, `countGte()`.
- **Tests**: 26 new tests covering all new ops including nested composites, while-clause invalidation with `or`, and suspended block resume through composite predicates.

### FIXED (implemented + tested)

**C10: Change tracking on MountResult**

- **Was**: No way for callers to see what changed during a mount without polling.
- **Now**: `MountResult.changes: PathChange[]` reports every path mutation (direct, cascade, invalidate, resume) with old/new values. Pull-based notification consistent with the generator model.
- **Tests**: 4 new tests covering direct changes, cascade changes, invalidation changes, and resume changes.

**C8: Compaction policies**

- **Was**: `compact()` kept last value per path, discarded everything. No per-path rules.
- **Now**: Policy `compact` field on paths: `'preserve'` (never compact), number N (keep every Nth block), `'default'` (last-value-wins). Set via `mount('policy', path, { compact: 'preserve' })`.
- **Tests**: 4 new tests covering preserve, snapshot_every, and mixed policies.

**C9: Branch-and-bound search**

- **Was**: `search()` picked the locally-best capability greedily. Missed globally-optimal plans.
- **Now**: Explores all matching capabilities at each step, prunes when cumulative probability drops below the best complete plan found. Best-first ordering. Returns globally optimal plan within depth limit.
- **Tests**: Existing 8 search tests pass (finds same or better plans).

### DOCUMENTED (intentional design)

**C5: Cascade is ordered, mutations visible to subsequent steps**

- Documented in sequence.ts header. Cascade processes one path at a time via BFS queue. Each derived value is computed against the CURRENT projection (including mutations from earlier cascade steps). This matches append-only semantics where later statements see effects of earlier ones. The visited set prevents infinite loops.
- If deterministic merge is needed for communities/datatypes, Option C (two-phase: compute all from pre-cascade state, then apply) can be revisited.

---

**C7: Weak predicate language in evalConstraint**

- **Problem**: `evalConstraint` handles 9 ops (eq, neq, lt, lte, gt, gte, exists, notExists, count_lt). 15+ files need richer predicates.
- **Missing**: Composite conditions (AND/OR), type predicates, hysteresis, regex on values, temporal windows.
- **Design question**: Should composite predicates be built by composing simple Constraints (e.g., an AND is an array of Constraints that all must pass), or should the Constraint type gain an explicit `and`/`or` op?
- **Note**: The `where` array already acts as AND (all must pass). What's missing is OR within a single where position, and nested composition.
- **Files affected**: watchers/ (all 6), deltat/ (temporal windows), logicandoperations/monitoring

---

**C8: Aggressive compaction**

- **Problem**: `compact()` keeps the last value per `op:path` key and discards everything. No per-path or per-subtree compaction policies.
- **What's needed**: Policy-driven compaction — "compact body edits but preserve section headers", "keep every Nth revision as an audit point", "never compact paths under `audit.*`".
- **Design question**: Should compaction policies be Constraints on types (declarative, inspectable), or runtime configuration on the Sequence?
- **Files affected**: eventeconomics/historicals, obligations/trace, narrativemodel/compaction, languagesemantics/compression-semantics

---

**C9: Greedy search algorithm**

- **Problem**: `search()` picks the locally-best capability at each step. No branching, no backtracking. Misses globally-optimal plans.
- **What's needed**: Branch-and-bound search, scenario branching (what-if), pruning when a branch's probability drops below threshold.
- **Design question**: Is search a method on Sequence, or a separate algorithm that reads the Sequence's projection? The latter is cleaner but harder to make incremental.
- **Files affected**: logicandoperations/branchbound, eventeconomics/planbranch, eventeconomics/simulations

---

**C10: No external notification mechanism**

- **Problem**: Cascade is internal. No way for external code to subscribe to path changes, gap fills, or while-clause breaks.
- **Current state**: Callers poll via `get()`, `gaps()`, `suspended()`. No push.
- **Option A**: Callback registration on Sequence (`onPathChange(path, fn)`). Simple but couples notification to the Sequence.
- **Option B**: MountResult already returns `cascaded`, `invalidated`, `resumed` — callers can react to these. Just needs better exposure.
- **Option C**: The Sequence emits to an event bus / observable. External code subscribes.
- **Recommendation**: Option B (improve MountResult) is sufficient for most use cases. Option C if watchers/ need truly decoupled observation.
- **Files affected**: watchers/ (all 6), distributedsched/ (heartbeating, lockleasing), streams/
