# Self-Render Semantics

A process generates its own prompt by reading its own state through the same interface that any external consumer would use. The prompt is not "built" by a separate prompt-construction system -- it is a projection of the process's typed state into text. Different consumers (LLM, UI, scheduler) see the same data in different formats, but the data is identical.

The hard part is attention management. A process has regions (system instructions, tools, state, history, input), each competing for token space in a finite context window. Regions have budgets, and when content exceeds a budget, the excess is compressed into expansion references the LLM can request to expand. Budget composition is monotonic -- refinement can only tighten budgets, never widen them.

## Problem Context

- **Actor(s)**: The process itself (reading its own state to generate a prompt), LLMs (consuming the prompt), UIs (rendering state visually), schedulers (reading state programmatically).
- **Domain**: Self-referential prompt generation from typed process state, with attention management across a finite context window.
- **Core Tension**: A process must render its own state as a prompt from the same data interface external consumers use, but a finite context window forces budget allocation, compression, and attention prioritization across competing regions.

## Requirements

**R1**: A process SHALL generate its own prompt by reading its own state through the same interface available to any external consumer.
- *Rationale*: A single data interface guarantees that no consumer (including the process itself) sees different data -- only different presentations.
- *Verifiable by*: The prompt generated by self-rendering contains the same data values as those returned by external reads of the same state.

**R2**: Process state SHALL be organized into named regions, each with a declared purpose, content, token budget, lock state, and mutation policy.
- *Rationale*: Named regions are the structural units for attention management within a finite context window.
- *Verifiable by*: Each region is identifiable by name and has all declared properties.

**R3**: When a region's content exceeds its token budget, excess content SHALL be compressed into expansion references -- content-addressable tokens that the LLM can request to expand.
- *Rationale*: Hard truncation loses information; expansion references preserve access while staying within budget.
- *Verifiable by*: A region with 1000 tokens of content and a budget of 500 renders 500 tokens plus expansion references for the remainder.

**R4**: Each expansion reference SHALL include a short identifier, a path back to the source data, a type signature of what expansion reveals, and an estimated token cost.
- *Rationale*: The LLM needs enough metadata to decide whether expansion is worthwhile before requesting it.
- *Verifiable by*: Expansion references contain all four properties; the LLM can use the identifier to request expansion.

**R5**: Budget refinement SHALL be monotonic -- a child region's budget can only be tighter (smaller) than the parent's allocation, never wider.
- *Rationale*: Monotonic budgets prevent child regions from claiming more space than allocated, which would break the overall context budget.
- *Verifiable by*: Attempting to set a child region's budget larger than the parent's allocation is rejected.

**R6**: Locked regions SHALL appear verbatim in the prompt and SHALL NOT be modifiable by the LLM or downstream refinement.
- *Rationale*: System instructions and other invariant content must persist unchanged across the entire conversation.
- *Verifiable by*: A locked region's content is identical in the rendered prompt to its declared content; no operation can alter it.

**R7**: The same process state SHALL render in different formats for different consumers (text prompt for LLM, terminal output for CLI, structured data for UI) with identical underlying data.
- *Rationale*: Multi-format projection ensures consistency across all consumers.
- *Verifiable by*: Rendering the same state in two different formats produces outputs that, when parsed, contain the same data values.

**R8**: The rendered prompt SHALL include a section listing all unresolved items with their paths and expected types.
- *Rationale*: The LLM needs to see what remains unfilled to decide what to work on.
- *Verifiable by*: Each unresolved item in the process state appears in the rendered unresolved-items section with its path and expected type.

**R9**: Each region SHALL declare a mutation policy (expand or compress) governing what transformations are permitted.
- *Rationale*: Some regions should be expandable (show more detail); others should be compressible (summarizable). The policy prevents inappropriate transformations.
- *Verifiable by*: A region with mutation policy "compress" can be summarized; a region with policy "expand" can be elaborated.

## Acceptance Criteria

**AC1** [R1]: Given a process with state values at paths A, B, C, when the process self-renders and an external consumer reads the same paths, then the data values are identical.

**AC2** [R2]: Given a process with regions "system", "tools", "state", and "history", when each is inspected, then each has a name, content, budget, lock state, and mutation policy.

**AC3** [R3, R4]: Given a region with 1000 tokens of content and a budget of 500, when rendered, then 500 tokens of inline content appear plus expansion references for the excess, each with an identifier, path, type signature, and estimated cost.

**AC4** [R5]: Given a parent allocation of 2000 tokens for a region, when a child attempts to set its budget to 3000, then the attempt is rejected.

**AC5** [R6]: Given a locked "system" region with content "You are agent-X in moment 11", when the prompt is rendered across multiple turns, then that content appears verbatim and unchanged.

**AC6** [R7]: Given a process state, when rendered as a text prompt and as structured data, then both contain the same underlying values.

**AC7** [R8]: Given a process with 3 unresolved items, when the prompt is rendered, then all 3 appear in the unresolved-items section with paths and expected types.

## Open Questions

- What is the compression algorithm for excess content? Is it LLM-generated summarization, structural elision, or something else?
- How are region budgets initially allocated across the context window? Is there a default allocation strategy?
